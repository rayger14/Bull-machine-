# Feature Engineering Pipeline Failure - Root Cause Analysis

**Investigation Date:** 2025-12-18
**File:** `data/features_mtf/BTC_1H_2022-01-01_to_2023-12-31_with_macro.parquet`
**Status:** 🔴 CRITICAL - Regime features completely broken for 2022-2023 data

---

## Executive Summary

The HMM regime detection pipeline is failing because **3 critical macro features are ALL ZEROS/NULLS** in the training data file. This is NOT a data source issue - it's a **pipeline execution bug** where features were merged from an incomplete macro history file.

**Impact:** HMM regime classifier cannot train properly - it needs RV_20, funding_Z, and proper data coverage.

## Root Cause Breakdown

### 1. **The Zero Features Problem**

```python

```

### 2. **Why They're Broken**

**Path 1: macro_history.parquet has NO 2022-2023 data**

```bash
# The source file is INCOMPLETE:
$ python3 -c "import pandas as pd;
macro = pd.read_parquet('data/macro/macro_history.parquet');
print('First non-null rv_20d:', macro[macro['rv_20d'].notna()]['timestamp'].min())"

Output: 2024-01-01 00:00:00+00:00  # ⚠️ Data only starts in 2024!
```

**Root Cause:** `macro_history.parquet` was built/updated in 2024 and never backfilled for 2022-2023 historical data.

**Path 2: Feature merge script blindly merged NULLs**

The file was generated by: `bin/append_macro_to_feature_store.py`

```python
# From append_macro_to_feature_store.py lines 100-106:
merged = pd.merge_asof(
    features,
    macro_filtered,
    on='timestamp',
    direction='backward',  # Use most recent macro value
    tolerance=pd.Timedelta(hours=2)  # Max 2-hour gap
)
```

**What Happened:**
1. Script loaded `macro_history.parquet` (which has NO rv_20d/funding before 2024)
2. merge_asof found NO matching macro data for 2022-2023 timestamps
3. Result: ALL rv_20d, funding columns became NULL
4. Parquet file saved with NULL values → displayed as zeros in analysis

### 3. **Why RV_20 is Special (and Confusing)**

There are **TWO different RV_20 features** in the file:

```python
# From column analysis:
RV_20:   0 non-zero values  # Computed feature (broken) ❌
rv_20d:  0 non-null values  # Raw macro feature (never populated) ❌

# But these WORK:
RV_7:    17,206 non-zero values  # Computed from TOTAL market cap returns ✅
RV_30:   16,775 non-zero values  # Computed from TOTAL market cap returns ✅
```

**Why the inconsistency?**

Looking at `bin/add_p1_features_to_macro.py` lines 140-150:

```python
# RV_7 and RV_30 are COMPUTED from TOTAL market cap:
if 'TOTAL' in df.columns:
    df['TOTAL_ret'] = df['TOTAL'].pct_change().fillna(0)
    df['RV_7'] = compute_realized_volatility(df['TOTAL_ret'], window=7*24)
    df['RV_30'] = compute_realized_volatility(df['TOTAL_ret'], window=30*24)

# But RV_20 is just RENAMED from rv_20d (which doesn't exist!):
if 'rv_20d' in df.columns:
    df['RV_20'] = df['rv_20d']  # ❌ This is NULL, so RV_20 = NULL → 0
    df['RV_60'] = df['rv_60d']  # ❌ Same issue
```

**The Bug:** RV_20 should have been computed like RV_7/RV_30, but instead it was just renamed from a non-existent column.

## Scripts Involved

### 1. **Primary Pipeline Script**
**File:** `bin/build_mtf_feature_store.py` (1,295 lines)

**Purpose:** Main feature engineering pipeline that builds the MTF (Multi-TimeFrame) feature store.

**What it does:**
- Loads OHLCV data for 1H, 4H, 1D timeframes
- Computes technical indicators (ATR, RSI, ADX, SMAs)
- Computes Wyckoff, BOMS, structure features
- Does NOT compute RV_20, funding_Z (expects them from macro file)

**Key insight:** This script does NOT compute realized volatility or funding features - it relies on them being added later via the macro append script.

### 2. **Macro Feature Append Script**
**File:** `bin/append_macro_to_feature_store.py` (148 lines)

**Purpose:** Merge macro features from `macro_history.parquet` into existing feature stores.

**What it does (lines 93-106):**
```python
# Merge macro features using asof merge (forward-fill behavior)
merged = pd.merge_asof(
    features,           # Feature store (2022-2023 data)
    macro_filtered,     # Macro history (has NULLs for 2022-2023!)
    on='timestamp',
    direction='backward',
    tolerance=pd.Timedelta(hours=2)
)
```

**The Bug:** When macro_filtered has NO DATA for the timestamp range, merge_asof returns NULLs for all macro columns. Script does not validate or warn about this.

### 3. **Macro Feature Computation Script**
**File:** `bin/add_p1_features_to_macro.py` (200+ lines)

**Purpose:** Add priority 1 (P1) features to macro_history.parquet including RV_7, RV_30, RV_20, etc.

**What it should do (lines 140-150):**
```python
# Compute RV from market cap returns
df['TOTAL_ret'] = df['TOTAL'].pct_change().fillna(0)
df['RV_7'] = compute_realized_volatility(df['TOTAL_ret'], window=7*24)
df['RV_30'] = compute_realized_volatility(df['TOTAL_ret'], window=30*24)

# BUG: RV_20 is just renamed, not computed!
if 'rv_20d' in df.columns:
    df['RV_20'] = df['rv_20d']  # ❌ rv_20d doesn't exist for 2022-2023
    df['RV_60'] = df['rv_60d']  # ❌ Same issue
```

**The Fix Needed:** Lines 146-150 should compute RV_20/RV_60 the same way as RV_7/RV_30:

```python
# FIX:
df['RV_20'] = compute_realized_volatility(df['TOTAL_ret'], window=20*24)
df['RV_60'] = compute_realized_volatility(df['TOTAL_ret'], window=60*24)
```

## Lessons Learned

1. **Always validate after merges** - `merge_asof` silently returns NULLs when no match found
2. **Feature computation vs feature merging** - RV_20 inconsistency shows confusion between computed and raw features
3. **Historical backfill is critical** - macro_history.parquet was built for 2024+ but training needs 2022-2023
4. **Test data coverage first** - Should have checked macro file date range before using it
